# /decomp-lead — Strategic roadmap maintainer

SCOPE = `$ARGUMENTS`

Reads project state across the splat yaml, `src/`, `nonmatchings/`, and Ghidra MCP, and rewrites `docs/decomp_roadmap.md`. Its job is to *suggest* the next 5 targets — not to dispatch them. You pick from the roadmap and invoke `/decomp <func>` yourself.

<scope>
This command modifies exactly one file: `docs/decomp_roadmap.md`. It never edits `mariogolf64.yaml`, `src/`, `symbol_addrs.txt`, `ghidra_symbols.txt`, `re_tracking/*.yml`, or any other state. Read-only against everything else.
</scope>

## Step 0 — Validate argument

If `$ARGUMENTS` is non-empty, anchor-check against `^[a-z][a-z0-9_-]*$`. On mismatch, abort with this exact one-liner:

```
/decomp-lead: arg `$ARGUMENTS` does not match `^[a-z][a-z0-9_-]*$`. Aborting.
```

Empty `$ARGUMENTS` is valid — it means "full project review".

## Step 1 — Acquire the MCP lock

```bash
tools/mcp_lock.py acquire --command decomp-lead --identifier ${ARGUMENTS:-full}
```

Non-zero exit means another `/decomp` or `/decomp-lead` already holds it — abort with `mcp_lock.py`'s stderr message. The lock survives across Bash tool calls (see `tools/mcp_lock.py --help`); Step 7 must end with `tools/mcp_lock.py release --identifier ${ARGUMENTS:-full}`. The release call must also run on any early-abort path (MCP down, write check failure, etc.).

Same flock as `/decomp` — there's only one Ghidra slot.

## Step 2 — MCP discovery

Same dance as `/decomp` (call `list_instances` → `list_tool_groups` → `connect_instance` defensively). Bail with the same helpful message if MCP is down. All MCP calls pass `program="baserom.z64"`.

## Step 3 — Read project state

In parallel, gather (read-only):

- `mariogolf64.yaml` — subsegment statuses (`asm` / `c` / `hasm` per subsegment).
- `src/*.c` — file roster + INCLUDE_ASM density gives matched/total per file.
- `nonmatchings/*/` — leftover scratch dirs. Directory presence + `base.c` mtime indicate recent activity.
- `build/mariogolf64.map` if present — function sizes.
- Ghidra MCP — call the discovered tools for `list_functions` / `get_callers` / `get_callees` / `decompile_function` (for spot-checks of candidate leaves).
- `~/development/reversing/ghidra/mariogolf64/re_tracking/{functions,structs,memory_segments,globals}.yml` — committed snapshots for cross-checks.
- `ghidra_symbols.txt` mtime + line count — for sync-status reporting.

## Step 4 — Compute matched-function count

<count-rules>
- **Total** = count of `glabel` occurrences across all `asm/*.s`. Includes everything: entry stubs, libkmc helpers, hasm modules.
- **Matched** = (functions in `c`-flipped segments whose `INCLUDE_ASM` line is gone, i.e. function body is inline) **plus** (all functions in `hasm` subsegments — they count as matched-by-permanent-asm).
- **Unmatched** = Total − Matched. Subdivides into:
  - "In c-flipped backlog" — `INCLUDE_ASM` still present in a `c`-flipped `src/<seg>.c`.
  - "Not yet started" — subsegment is still `asm`.
- **File-level md5 candidates** = `c`-flipped files with zero `INCLUDE_ASM` lines.
</count-rules>

## Step 5 — Score and rank candidates

Apply these heuristics to every unmatched function (skip those in `hasm` subsegments):

<scoring>
- **Leaf-first**: +10 if all callees are matched (or excluded as `hasm`), or if no callees.
- **Size band**: +5 if the function size is in [20, 200] instructions (quick wins).
- **File continuity**: +3 if the parent `src/<seg>.c` is already partially matched — finishing files yields md5-checkable wins.
- **Subsystem coherence**: +1 if it sits inside `$SCOPE`'s subsystem (per Ghidra MCP call-graph adjacency).
- **No Ghidra name**: −2 (placeholder-only functions are usually harder to reason about).
- Sort: score desc, then by vram asc (deterministic — roadmap diffs stay clean).
</scoring>

If `$SCOPE` is non-empty, filter the candidate list to that subsystem before sorting.

## Step 6 — Rewrite `docs/decomp_roadmap.md`

Use exactly this section template. All counts are integers; percents are two decimals; the timestamp is `YYYY-MM-DD HH:MM TZ`.

```markdown
<!-- Generated by /decomp-lead. Do not edit by hand. -->
# Decomp roadmap

_Last updated: <timestamp>_  _Scope: <SCOPE or "full project">_

- Functions: `<matched>/<total>` (`<percent>%`)
  - C body inline: `<inline>`
  - hasm (permanent): `<hasm>`
  - In c-flipped backlog: `<backlog>`
  - Not yet started (asm): `<notstarted>`
- File-level md5 candidates: `<file-candidates>/<file-count>`

## Strategy

<3–5 sentences explaining the current phase and the rationale for the next-5 picks. Reference call-graph structure, recent successes, dependencies. No filler.>

## In flight

| File | Matched/Total | Notes |
| --- | ---: | --- |
| `src/<seg>.c` | `<m>/<t>` | <stuck-function or recent-activity note, if any> |
| ...           |             |   |

## Next 5 targets

1. `func_name` (size, parent_file, caller chain). Rationale: <one line>.
2. ...

## Subsegments to flip (`asm` → `c`)

Cohesive groups ready to be opened (callees mostly matched, structs known):

- `<offset_or_seg_name>` — <one-line rationale>
- ...

## Sync status

- `ghidra_symbols.txt`: <line_count> entries (mtime: <date>).
- Struct parity: <decomp_count> headers vs <ghidra_count> in `re_tracking/structs.yml`. Diff: <names_only_in_decomp>, <names_only_in_ghidra>.
- Memory-map drift: <count> mismatches between `mariogolf64.yaml` subsegment boundaries and `re_tracking/memory_segments.yml`. First 3:
  - <vram start> ... <vram end> — <reason>
```

If `docs/` does not exist yet, create it. Atomic-overwrite the roadmap file. Verify post-write that `git status` reports only `docs/decomp_roadmap.md` as changed; abort with a diagnostic if anything else moved.

## Step 7 — Final report

Release the lock, then emit one short message in the chat (per `PROMPT_GUIDELINES.md` — concise, no preamble):

```bash
tools/mcp_lock.py release --identifier ${ARGUMENTS:-full}
```

```
Roadmap updated: docs/decomp_roadmap.md

Top recommendation: <func_name> — <one-line why>.
Run `/decomp <func_name>` to start.
```

That's it. The lead never invokes `/decomp` itself.
